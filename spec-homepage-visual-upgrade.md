# 主页视觉升级规格说明

## 背景

当前首页已经具备配置驱动、插件渲染、主题切换、入场动画、Progress Rail、工具过滤、项目展开和联系抽屉等功能。视觉问题不在功能完整性，而在首页缺少获奖级作品常见的叙事结构、版式主张和内容策展。

本规格基于当前桌面 1440x900 与移动 375x812 首屏检查，并参考 Awwwards、CSS Design Awards、Web Design Awards 等获奖网站索引与评审维度形成。参考方向仅用于提炼设计原则，不复制具体作品视觉。

参考源：

- Awwwards Winning websites: https://www.awwwards.com/websites/
- CSS Design Awards: https://www.cssdesignawards.com/
- Web Design Awards judging rubric: https://www.webdesignawards.io/

## 当前问题

- 首屏以头像、姓名、两个按钮居中排布，信息表达过少，无法快速说明“我是谁、我做什么、为什么值得继续看”。
- 桌面首屏下半部直接露出 Navigation 模块，但搜索框和标题分散，未形成有意图的构图。
- 移动端首屏存在大段空白，用户需要下滑才看到个人定位和资源入口。
- 视觉语言过度依赖深色玻璃态、霓虹蓝紫、圆形头像和卡片网格，容易显得模板化。
- Profile 的 `bio`、`title`、`skills`、`location` 等配置字段在首屏没有被充分使用。
- Navigation、Tools、Projects 的模块层级过于平均，缺少“精选作品/核心能力/日常工具”的优先级。

## 设计目标

- 让首页第一屏形成明确身份叙事：姓名、职业定位、能力范围、可信信号、下一步行动。
- 把首页从“模块列表”升级为“可滚动的编辑作品”，每个 section 有不同节奏和清晰主次。
- 保持纯静态、配置驱动、插件化架构，不引入服务端依赖。
- 控制视觉资产和动效成本，避免为了高级感牺牲性能、可访问性和内容真实性。
- 在桌面与移动端分别形成自然构图，而不是移动端只是桌面的压缩版。

## 非目标

- 不克隆任一获奖网站的布局、文案、视觉资产或交互。
- 不引入 WebGL、Three.js、GSAP 等重型动效库作为第一阶段依赖。
- 不使用假统计、假履历、假客户、假奖项或不存在的作品图。
- 不把所有插件改成营销落地页式内容。
- 不为了视觉效果破坏现有配置契约与自动化质量门禁。

## 设计原则

### 1. Editorial Hero

首屏应像作品集封面，而不是资料卡。

必备内容：

- 姓名：作为主视觉标题，但不必永远居中。
- 职业定位：使用 `profile.title`。
- 一段 1-2 行的个人叙事：使用 `profile.bio`。
- 可信信号：位置、核心技能数量、部署/工程质量等真实信息。
- 主要行动：联系我。
- 次要行动：查看项目。

桌面建议：

- 使用非对称 12 栏构图，左侧为大字叙事，右侧为头像/身份面板/能力索引。
- 首屏高度控制在 `min-height: calc(100vh - header)`，但底部必须露出下一 section 的提示。
- 保留头像，但降低“圆形发光头像”的模板感，可改为方形裁切、窄边框、编号标签、坐标信息或签名式排版。

移动建议：

- 姓名、职业定位、简介和按钮必须在首屏内完整可见。
- 头像尺寸收敛，不占据过多高度。
- CTA 可垂直排列，但间距必须紧凑，避免首屏大空洞。

### 2. Curated Navigation

Navigation 不应只是链接卡片集合，应改成“资源目录/快速入口”的策展区。

要求：

- 搜索框与标题必须属于同一个视觉组，不允许在桌面首屏两端孤立。
- 分类标签应支持横向扫描，当前项目内容少时避免三列稀疏卡片。
- 站内入口与外部资源在视觉上区分。
- `mailto:` 和 `#anchor` 不打开新标签的规则保持不变。

### 3. Tool Matrix

Tools 应从普通卡片网格升级为能力矩阵。

要求：

- 工具卡片应展示类别、名称、用途，避免随机生成“专家/架构级”等不可验证标签。
- `favorite` 可表现为重点工具，但不能暗示虚假熟练度。
- 分类过滤继续保留，控件应更像紧凑 segmented control。
- 2-4 个工具时桌面不应铺成稀疏四列，可采用紧凑矩阵或横向分组。

### 4. Featured Project

Projects 当前只有一个项目，三列卡片不合适。

要求：

- 当项目数量为 1 时使用 featured case-study 布局。
- 展示项目目标、技术栈、质量门禁、部署方式和可展开详情。
- 项目数量大于 2 时再切换到网格。
- 不使用不存在的项目截图；如无图片，用代码片段、结构图或抽象工程面板替代。

### 5. Visual System

从“TECH_PRO 深色模板”升级为“Editorial Engineering”。

建议令牌：

- 背景：接近黑但带细微暖灰，不使用大面积纯黑。
- 主文字：高对比暖白。
- 辅助文字：低饱和灰。
- 强调色：保留青色，但只用于状态、焦点和少量高亮。
- 第二强调色：引入少量暖色或中性色，用于平衡蓝紫单一感。
- 玻璃态：降低使用频率，优先使用实色面板、细边框和排版层级。

约束：

- 不使用大面积紫蓝渐变作为主背景。
- 不使用装饰性 orb、bokeh blob 或无语义发光团。
- 卡片圆角控制在 8px 以内，除头像或图标容器外避免大圆角。
- 文字不使用负 letter-spacing。

### 6. Motion

动效应服务于注意力，而不是持续漂浮。

要求：

- 入场动画保留，但与新视觉语言一致。
- 头像不默认无限浮动；如保留动画，必须尊重 `prefers-reduced-motion`。
- section reveal 使用轻微位移和透明度即可。
- 鼠标背景效果降低存在感，避免抢首屏信息层级。

## 信息架构

推荐顺序：

1. Profile Hero：身份叙事、能力信号、CTA。
2. Featured Project：先展示最有代表性的项目。
3. Tools Matrix：展示工作流能力。
4. Curated Navigation：放站内/外部资源入口。
5. Social / Contact：轻量收尾。

说明：当前插件顺序是 Profile、Navigation、Tools、Social、Projects。视觉升级阶段建议把 Projects 提前到 Tools 前，Social 放到末尾；该调整通过 `plugins.config.json` 的 `order` 完成，不需要破坏插件系统。

## 实施阶段

### Phase 1：Hero 与视觉系统

- 重构 Profile 插件首屏布局。
- 使用 `title`、`bio`、`skills`、`location` 字段。
- 调整全局色彩、圆角、间距、文字层级。
- 降低玻璃态和霓虹效果。
- 优化移动端首屏信息密度。

### Phase 2：模块策展

- Projects 支持单项目 featured 布局。
- Tools 移除不可验证熟练度文案，改为工具用途矩阵。
- Navigation 搜索区与分类区重排。
- 调整插件默认顺序。

### Phase 3：动效与验收

- 调整 IntroStage 与新视觉语言一致。
- 优化 reveal 节奏与 reduced-motion 降级。
- 用 Playwright 覆盖桌面和移动首屏视觉结构。
- 补充 320、375、768、1024、1440 响应式人工验收。

## 验收标准

- 1440x900 首屏内可见姓名、职业定位、简介、联系按钮、查看项目按钮、至少 3 个真实身份/能力信号。
- 375x812 首屏内可见姓名、职业定位、简介和至少一个主要 CTA，不出现超过 160px 的无意义垂直空白。
- Profile Hero 不再只由头像、姓名、两个按钮构成。
- 项目数量为 1 时不使用三列网格卡片布局。
- Tools 不展示“专家”“架构级”“精通”等由索引随机生成的熟练度标签。
- Navigation 的搜索框不在桌面首屏孤立于标题之外。
- 页面仍通过 `npm run lint`、`npm run check:assets`、`npm run build`、`npm run test:e2e`。
- 页面仍不展示 `example.com`、`yourusername`、假统计、假评论、假下载。
- `prefers-reduced-motion: reduce` 下无无限浮动、扫描或强位移动画。
- Lighthouse 可访问性目标保持 90+，普通正文颜色对比满足 WCAG AA。

## 风险

- 视觉升级可能改变 Playwright 文案定位，需要同步测试选择器。
- 如果继续只有一个项目和很少外部资源，设计需要更偏“个人叙事 + 工程说明”，不能伪造内容撑版面。
- 若引入新的图像素材，必须走资源检查，且不能依赖远端图片。
